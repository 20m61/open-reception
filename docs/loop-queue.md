> （下記「AC 締めループ」）。新規ビルドの重複を避けること。

## AC 締めループ（各周回の冒頭に追加）

1. `gh issue view <N>` で受け入れ条件（AC）を列挙する。
2. 既存コード/テスト（ルート・コンポーネント・ドメイン・E2E）へマッピングする。
3. **未充足の AC だけ**を increment として実装する（TDD・純ロジック先行）。
4. ローカル品質ゲート → PR → セルフレビュー → ユーザー確認 → squash マージ。
5. 全 AC 充足なら issue をクローズし、親 epic のチェックボックスを更新する。
