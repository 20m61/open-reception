## 3. 各成果物の引き継ぎ事項

### A. #365 音声評価ハーネス（PR #393）— #369〜#372 の土台

`docs/voice-evaluation-harness.md` が契約ドキュメント。**#369〜#372 はこのイベント形式に
計測値を出し、`validateVoiceEvalSession(session).errors` が空であることを適合ゲートとする。**

確定した設計判断（レビュー 3 巡で固めた。**変更は高くつく**）:
- **正解ラベルは刺激側**（`nearEndStimuli[]` の `{atMs, toleranceMs, label}`）に固定し、観測 onset を
  窓でマッチングする。観測通番に紐づけると、VAD が onset を 1 つ取りこぼしただけで以降が全てずれ、
  **provider の回帰が「スキーマ違反」に化けて「実装の壊れ」と「計測の壊れ」が区別できなくなる**。
- **計測不能は `null`**（分母 0 を満点にしない）。`strict: true` プロファイルでは欠落自体を違反にする。
- **帰属できない barge-in 停止を消さない**。`unattributedBargeInStops` が悲観的遅延で計上し、
  `maxUnattributedBargeInStopRate: 0` を両プロファイルに設定。これが無いと、検出率の余裕
  （ci で 10%）が見逃しを吸収して**任意に大きい停止遅延がゲートを通り抜ける**。
- **tolerance は 300ms**（uat の barge-in 停止 P95 と同値）。`consecutive-near-end` の刺激間隔を
  400 → 1200ms に広げて確保した。`onsetLagMs` ノブで崖がテスト固定されている
  （lag 250ms → 全緑、400ms → 検出 0 + 誤検出）。**実 provider の実測 lag をこれと突き合わせること。**

**未解決として文書化した曖昧さ**: 割り込み@1000 → 相づち@1200 → 停止@1500 は相づちに帰属し
300ms と報告される（真値 500ms）。イベントのみの窓規則では原理的に解けず、正解ラベルで
曖昧性を解くと誤停止率が測れなくなる。検出 0 / 誤停止 1 で赤くはなる。

### B. #373 階層組織（PR #394）— **#373 はオープン継続**

increment 1 は domain 層のみ。**additive 限定**で既存 `Department` / `staff.departmentId` は無改変
（変更/削除 0 行を確認済み）。残 increment の推奨順:
**永続化 repository → Directory API（`requireActor` から `OrganizationScope` を組む）→ 来訪者 UI 配線 → tenant 越境 E2E**。

**#374 が守るべき前提**:
- **fallback は organization モジュールに存在しない**。`listCallableMembers` は指定組織の所属のみを
  返し親子に遡らず、`ancestorsOf` は表示専用、`staff.fallbackStaffIds` も `acting` へ昇格させていない。
  → **RoutingPolicy 側が唯一の fallback 定義**になる（issue の「親子関係と取次 fallback を同一視しない」に忠実）。
- `siteId: undefined` は「テナント横断組織」を意味し、**親が横断なら子はどのサイトでも可**という
  非対称な境界規則。
- `OrganizationScope` は判別可能ユニオン（`{kind:'tenant'} | {kind:'site'}`）。site の埋め忘れを型で防ぐ。
- `listCallableMembers` は `now` が**必須**。`acting` に `isActiveAt` を適用し期限切れ代理を除外する。
  `isActiveAt` は `Date.parse` 比較で、**解釈不能値は fail-closed**（壊れた期間指定で代理が無期限
  有効化される方が危険なため）。

### C. #396（#394 follow-up）— **#374 の前に潰す**

- `buildOrganizationTree` の「防御的な回収」ブロック（`hierarchy.ts` L311-337）は**デッドコードかつ、

- `AffiliationQuery.scope` / `resolveActingMembers` の scope が**任意のまま**で、省略すると
  tenant/site 境界が漏れる（`toVisitorAffiliations` は公開性の境界は守るが越境は守らない）。
- `toVisitorOrganization` の `publicIds` が任意で**安全でない側が既定**（非公開の親 id が漏れる）。

## 4. 運用・環境の申し送り

### ループ運用で確立したこと
- **`/issue-ac-mapping` を各周回の冒頭で必ず通す**（skill 化済み）。「issue 本文とキューの分類は
  仮説であって事実ではない」— 過去 3 回、stale 分類で既存実装を作り直しかけた。
- **レビューは「実際に実行して実証」させる**。今回 3 巡すべてで、読解では見つからない欠陥が
  実測で出た（何もしない provider が緑 → マッチングの横取り → 修正が作り込んだサンプル削除）。
  ミューテーションテスト（旧バグを故意に再導入してテストが落ちるか）も有効だった。
- **甘い assertion がバグを通す**。`expect(ids.length).toBeGreaterThan(0)` や
  `expect(cer.p50 < 0.3)`（実装が無条件に 0 を返しても pass）が実際に見逃しを生んだ。値を固定する。

### ローカル環境固有（**web の Claude では事情が異なる**）
- **worktree のゲートは、その worktree 自身の `scripts/quality-gate.sh` を叩く**。スクリプトが
  `cd "$(dirname "$0")/.."` するため、main の絶対パスを渡すと **main が検証され worktree の変更は
  一切見られない**。今回これで 2 トラック空振りさせた（`docs/loop-workflow.md` に注記済み）。
- **この開発機は 2 コア / 4 スレッド**。ゲートの並行実行で load が 100 超まで飛ぶ。**直列で回すこと。**
  他プロジェクト（salon-loop 等）の vitest が走っていると更に詰まる。
- **並行トラックで他トラックのプロセスを kill しない**。同名の `npm run typecheck` が複数走るため、
  pid の見当で kill すると他トラックのゲートを壊す（今回実際に発生）。
- **コミット署名**: 1Password `op-ssh-sign`。本セッション終盤で `op` CLI は動くのに `op-ssh-sign` だけが
  「No SSH private key found」で失敗した（鍵は Personal vault にあり公開鍵も一致、アプリも起動中）。
  ユーザー判断で **PR #393/#394 の中間コミットは `-c commit.gpgsign=false` で未署名**にした
  （`--no-verify` は使っていないので pre-commit フックは通っている）。squash マージ後の main 上の
  コミットは GitHub 生成なので影響は中間コミットのみ。
- Dependabot が high 2 件を報告している（push 時に表示）。未対応。
