# カスタマージャーニー（理想論）

open-reception を「会社の顔」として運用するときの**理想的な**体験を、関係ロールごとに
end-to-end で描く。ここでは現状実装の制約ではなく**あるべき姿**を基準に書く（理想論）。
理想と現状の差分は各ジャーニー末尾の「現状ギャップ」に記し、`tests/e2e/journey-*.spec.ts` の
シナリオテストは**この理想に到達できている部分**を検証し、未到達は `test.fixme`/メモで可視化する。

関連: `docs/reception-ux-contract.md`（受付UX契約）, `docs/reception-issuance-design.md`（発行/
エンロール）, `docs/kiosk-integration-design.md`, `docs/multitenant-design.md`（ロール/境界）。

## 登場ロール

| ロール | 立場 | 主な画面 |
| --- | --- | --- |
| **Platform 運用者** (`developer`) | SaaS 提供側。テナント全体を横断管理 | `/platform/**` |
| **テナント管理者** (`tenant_admin` / `site_manager`) | 導入企業の受付運用責任者 | `/admin/**` |
| **受付端末** (Device/Kiosk) | 設置された iPad 端末そのもの | `/kiosk/enroll` → `/kiosk` |
| **来訪者** (Visitor) | 受付に来る人 | `/kiosk` |
| **担当者** (Staff) | 呼び出される社員 | `/staff/calls/[id]`・`/admin/staff` |

---

## ゴールデンパス（ライフサイクル全体）

```
Platform運用者: テナント発行
   └─ テナント管理者: ログイン → 拠点/部署/担当者/ブランド設定 → 受付端末の「受付URL/QR」を発行
        └─ 受付端末: QR/URL を開く → エンロール → 「会社の顔」として常時待機
             └─ 来訪者: 近づく → 用件を選ぶ → 担当者/部署へ即時接続 → 完了
                  └─ 担当者: 呼び出しを受ける → 応答 → 来客対応
```

理想は「**設置者は迷わず発行でき、端末は一度かざすだけで稼働し、来訪者は最短手数で正しい人に
繋がり、個人情報は端末に残らない**」こと。

---

## J1. Platform 運用者 — テナントを迎え入れる

**理想**: 新規導入企業を、安全な境界（テナント分離）付きで数分でオンボードでき、稼働状況を
一目で把握できる。

1. `/platform` ダッシュボードで全テナントの健全性（稼働端末数・異常）を俯瞰する。
2. `/platform/tenants` で新テナントを発行し、最初の `tenant_admin` を招待する。
3. 以降はテナント内部の個人情報に踏み込まず、**集計のみ**で運用を見守る（越権しない）。

**成功条件**: テナント越境ができない / 集計に PII が出ない / developer のみ platform 到達可。

**現状ギャップ**: テナント新規発行 UI は一覧表示中心（作成フローは後続）。招待導線は #70 系。

---

## J2. テナント管理者 — 受付体験を仕立て、端末を発行する

**理想**: 受付の「見た目・繋ぎ先・喋り方」を自社向けに整え、設置する端末ぶんだけ**受付URL/QR を
発行**して現場に配るだけで設置が完了する。危険操作は必ず確認と監査が伴う。

1. `/` (LP) の「ログイン」から `/admin/login` → ログイン。
2. `/admin/sites` で拠点（本社受付など）を確認・追加。
3. `/admin/departments`・`/admin/staff` で繋ぎ先（部署・担当者）を整える。在席状態も管理。
4. `/admin/branding` でロゴ・アクセント色・社名を入れ、待機画面を「会社の顔」にする。
5. `/admin/devices` で対象端末を選び **［受付URLを発行］** → モーダルに **URL＋QR＋有効期限**。
   「閉じると再表示不可」の注意を読み、QR を現場へ。失敗時は必ず理由が表示される。
6. 必要に応じて `/admin/reception-flows`・`/admin/call-routes`・`/admin/ai-guidance` で
   応対シナリオを調整。`/admin/audit` で誰が何を発行/失効したか追える。

**成功条件**: 発行は site-write 認可者のみ / 平文トークンは一度きり表示で監査に残らない /
発行失敗が無反応にならない / 旧 URL は再発行・失効で無効化される。

**現状ギャップ**: ブランド/フロー編集は実装済。発行は inc1。`/kiosk` の完全な session 必須化は
後続（`docs/reception-issuance-design.md` §6）。

---

## J3. 受付端末 — 一度かざすだけで「会社の顔」になる

**理想**: 設置者が QR をかざす（URL を開く）だけでエンロールが完了し、以後は再起動しても
PIN 不要で待機画面へ復帰する。配布済み URL の使い回しや期限切れは安全に弾かれる。

1. iPad で受付URL/QR を開く → `/kiosk/enroll` が「準備中…」を表示。
2. 使い捨てトークンを kiosk セッションへ交換し、`/kiosk` の待機画面（ヒーロー）へ。
3. 端末を再起動/ブックマーク再訪しても、**既存セッションを検知して**そのまま待機画面へ復帰
   （使い捨てトークンを再消費して締め出されない）。
4. 期限切れ・使用済み・無効端末の URL は明確なエラーと「管理画面で再発行」を案内。

**成功条件**: 単回性（同一 URL の二重利用を拒否）/ 再起動で締め出されない / revoke 後に旧 URL が
復活しない / 端末側に秘密が残らない。

**現状ギャップ**: 一覧の kioskId↔deviceId 完全統合は後続。consume の原子性は永続化増分で CAS 化。

---

## J4. 来訪者 — 最短で正しい人に繋がる（個人情報は残さない）

**理想**: 端末に近づくと迎えられ、用件を選ぶと正しい担当者/部署へ即時に繋がる。予約があれば
QR でさらに速い。呼び出しの結果（応答/不在/失敗）に応じて詰まらず代替に流れ、終わると痕跡なく
次の人のために待機へ戻る。

1. **待機**: アバター/ブランドのある待機画面。無操作でも警告は非ブロッキング。
2. **用件選択**: 「打合せ」「配送」等のクイックアクション（タッチファースト）。
3. **繋ぎ先**: 担当者を検索/選択、または部署を選択。不在者は選べず案内が出る。
4. **確認**: 氏名等を入れ、確認画面で内容を見てから明示的に「呼び出す」。
5. **結果**: 接続→対応 / 未応答→代替導線 / 失敗→やり直し。いずれも待機へ自動復帰。
6. **予約 QR**: 予約済みなら「QRで受付」で確認画面に直行（誤読・期限切れは確認操作で吸収）。

**成功条件**: 確認前に呼び出さない / 不在者は呼べない / 完了後に氏名（PII）が残らない /
無操作で待機へ戻り氏名が消える / 多言語・アクセシブル（critical a11y 違反ゼロ）。

**現状ギャップ**: 実カメラ QR デコードは後続増分（mock scanner）。

---

## J5. 担当者 — 呼び出しに気持ちよく応答する

**理想**: 自分宛の呼び出しが即座に届き、ワンタップで応答/折り返しでき、在席状態を自分で
切り替えられる。来訪者の PII は最小限しか見えない。

1. 呼び出しを受ける（通話/通知）。`/staff/calls/[id]` で来客の要点（用件・所要）を確認。
2. 応答アクション（接続/代替/完了）を選ぶ。
3. `/admin/staff` で自分を在席/不在に切り替える。不在中は受付画面で選べなくなる。

**成功条件**: 不在の即時反映 / 表示は運用に必要な最小情報 / 応答結果が監査に残る。

**現状ギャップ**: 実通話（Vonage）はコード実装済・実認証情報での疎通は #65。
