# iPad キオスク運用・実機 UAT チェックリスト (issue #8)

受付端末（iPad）を半常設で安定運用するための、端末設定・権限・ネットワーク・復帰動作の
実機検証手順。開発時の自動テスト（Playwright e2e）とは責務を分け、本書は **現場の実機 UAT** を対象とする。

## 0. 責務の分離

| 観点 | 担当 | 手段 |
| --- | --- | --- |
| 画面遷移・状態分岐・レイアウト崩れ | 開発 | `npm run test:e2e`（iPad viewport / 横縦） |
| 実機の権限・スリープ・通信復帰・常設運用 | 現場 UAT | 本チェックリスト（実機 iPad） |

自動 e2e で担保済みの分岐（成功/未応答/失敗/代替導線/完了後リセット）は、UAT では「実機で再現するか」を確認する。

## 1. 推奨条件

- 端末: iPad（第7世代以降を推奨。受付は横向き運用を初期推奨）
- OS: iPadOS は可能な限り最新のメジャー版
- ブラウザ: Safari、またはホーム画面に追加した PWA 表示
- 電源: 常時給電（受付完了後も画面を維持するため）
- ネットワーク: 安定した Wi‑Fi。固定 SSID を推奨

## 2. 画面向き・レイアウト

- [ ] 横向きで待機・目的選択・担当者/部署選択・入力・確認・呼び出し中・結果の各画面が見切れない
- [ ] 縦向きでも主要操作が可能（ボタンが小さすぎない、スクロールで到達できる）
- [ ] ソフトウェアキーボード表示時に入力欄が隠れない（来訪者情報入力）
- [ ] 画面回転で致命的に崩れない

## 3. ホーム画面追加 / PWA 運用

実装: `src/app/manifest.ts`（Web App Manifest）・`src/app/icon.tsx` / `src/app/apple-icon.tsx`
（アプリアイコン。テナントロゴ `/admin/branding` #88 とは独立したプロダクトアイコン）
（issue #331）。自動テストは manifest のフィールド・アイコンの生成ロジック・タブタイトルの
解決ロジックをカバーする（`src/app/manifest.test.ts` 等）。実機での見え方確認は本節が対象。

### 3.1 ホーム画面への追加（インストール）

- [ ] Safari で受付 URL（`/kiosk`）を開ける
- [ ] 共有シートの「ホーム画面に追加」を開くと、プロダクトアイコン（テナントロゴではなく
      共通の抽象マーク）とアプリ名「open-reception」がプレビューされる
- [ ] 追加したホーム画面アイコンの見た目が欠けたり潰れたりしていない（マスカブルアイコンの
      セーフゾーン確認。角が欠けても中心の図柄が残っていれば OK）

### 3.2 起動（ランチ）

- [ ] ホーム画面アイコンをタップすると `/kiosk`（受付待機画面）から起動する
      （manifest の `start_url`。#30 の長期セッション用 cookie には依存しないため、
      端末認可済みであればブラウザで開いた時と同じ画面に入れる）
- [ ] 起動直後、アドレスバー等のブラウザ chrome が表示されない（standalone 表示）
- [ ] 起動直後のステータスバー/背景色が突然真っ白にならない（`theme_color`/
      `background_color` がページ背景と揃っているため、起動アニメーション中の見え方が
      不連続に感じられないか確認する。iOS は明示的な apple-touch-startup-image が無くても
      アイコン + `background_color` から簡易スプラッシュを自動生成するため、まずはこの
      自動生成の見え方を確認する。専用スプラッシュ画像の作成要否は本確認結果を踏まえて
      別途判断する）
- [ ] 再起動後にホーム画面アイコンから受付画面へ復帰できる

### 3.3 待機画面への復帰（アイドル → 復帰）

- [ ] ホーム画面アプリを一定時間放置後、画面ロック解除や別アプリからの復帰で
      受付待機画面にそのまま戻る（白画面・クラッシュ・再ログイン要求が発生しない）
- [ ] マルチタスク画面から再度開いても、入力途中だった来訪者情報が残っていない
      （§9 プライバシーの完了後リセットと同じ確認観点をアイドル復帰後にも適用する）
- [ ] ホーム画面アプリのタブ/ウィンドウ切り替え上で、通常の Safari タブと視覚的に区別できる
      （standalone 表示・アイコンの効果）

### 3.4 ブラウザタブでの複数画面確認（管理画面）

- [ ] 管理画面（`/admin/*`）を複数タブで開いたとき、タブタイトルが「受付履歴 |
      open-reception」のように画面ごとに区別できる（テンプレート `%s | open-reception`）

## 4. ガイドアクセス / キオスク運用

- [ ] ガイドアクセス（設定 → アクセシビリティ → ガイドアクセス）で受付アプリに固定できる
- [ ] ガイドアクセス中にホームボタン/ジェスチャで離脱されない
- [ ] 画面消灯（自動ロック）を「なし」に設定、または常設運用に合わせて調整
- [ ] MDM を使う場合は単一アプリモード相当の設定を確認

## 5. アクセス許可（PIN / IP / 端末認可）

実装: PIN は `/admin/security`、端末登録/失効は `/admin/kiosks`（#18, #23, #29）。

- [ ] PIN 必須を有効にした場合、初回に PIN 許可画面が出る
- [ ] 正しい PIN で受付画面に進める
- [ ] 一度許可した iPad はリロード/ブラウザ再起動/iPad 再起動後も受付画面に復帰する（長期 kiosk セッション）
- [ ] 管理画面から端末を失効すると、受付画面が「利用停止」表示になる
- [ ] IP 許可リスト設定時、許可外ネットワークから許可されない
- [ ] 緊急停止モードを有効にすると全端末が受付停止になり、解除で復帰する

## 6. 権限（マイク / カメラ / 音声再生）

MVP はタッチ操作で完結し、音声/マイク/カメラが無くても受付が成立する方針。

- [ ] マイク権限を拒否しても受付フローが完走する
- [ ] カメラ権限を拒否しても受付フローが完走する
- [ ] 音声自動再生がブロックされても画面案内で受付できる（初回タップ後に音声有効化する設計を想定）

## 7. ネットワーク断 / 復帰

- [ ] 受付中に Wi‑Fi を切断しても画面が真っ白にならない（取得失敗時は表示継続）
- [ ] 呼び出し時に通信失敗すると失敗/代替導線へ遷移する
- [ ] ネットワーク復帰後、再操作で受付を継続できる
- [ ] 設定変更（部署/担当者/端末/セキュリティ）後、受付端末側に反映される（ディレクトリ/設定は API 取得）

## 8. スリープ / 長時間表示復帰

- [ ] スリープ復帰後に受付画面が表示される
- [ ] 長時間（1 営業日以上）の連続表示でクラッシュ・操作不能が起きない
- [ ] 一定時間操作が無いと待機画面へ戻る（自動リセット）

### 8.1 iPad 実機 1 営業日 soak チェックリスト (issue #317, charter 成功指標)

PROJECT_CHARTER の成功指標「受付端末として 1 営業日以上の連続稼働テストに耐える」を、
**実機**で検証するためのチェックリスト。ブラウザ自動化で担保できる範囲（メモリ増加率・
DOM ノード数・heartbeat 継続・console エラー・時刻加速による境界検証）は
`npm run test:soak`（`tests/e2e/soak/`、`docs/quality-gate.md` 参照）が既に自動でカバーする。
本節はそれでも自動化できない**実機固有の要因**（実 Safari のメモリ管理・GPU/WebGL コンテキスト
ロスト・実 Wi-Fi の断続・端末スリープ/ガイドアクセスとの相互作用・実際の 1 営業日という
時間経過そのもの）を対象にする。

**実施自体は #65（実機/実認証/実アセットが要る検証のスタック issue）に積み、実施者・実施日は
本チェックリストの実施時に issue コメントで記録する（下記「記録先」参照）。** 本チェックリスト
自体は実施の準備として今から整備しておくが、自動化できない以上「実施を要する」ことを issue 化
することで、担当者が確保でき次第、着手できるようにする。

**前提**:

- 対象は実運用と同等の設置（常時給電・固定 Wi-Fi・ガイドアクセスまたは MDM 単一アプリモード）。
- 開始前に `npm run test:soak`（smoke モード）が green であることを確認する（実装側の既知の
  リグレッションが無い状態で実機 soak に臨む）。

**手順**:

- [ ] 始業時刻に受付画面（`/kiosk`）を表示し、開始時刻・iPadOS/Safari バージョン・端末個体を記録する
- [ ] 業務時間中（目安 8 時間以上）、通常運用相当の頻度で受付フロー（待機→目的選択→担当者/部署
      選択→入力→確認→呼び出し→結果→完了）を反復させる（実来訪者の operating、または要員による
      模擬操作）
- [ ] 途中、意図的に Wi-Fi を数分切断し復帰させる（現場の AP 再起動や電波不安定を模す）。
      オフライン表示が出て、復帰後に自動で通常表示に戻ることを確認する
- [ ] 途中、画面ロック/スリープを数回発生させ、復帰後に受付画面が白画面/操作不能にならないことを
      確認する
- [ ] 途中、ガイドアクセス/MDM 単一アプリモードからの意図しない離脱が起きないことを確認する
- [ ] アバター/VRM を有効にしている場合、長時間表示後も描画が乱れない（WebGL コンテキスト
      ロストで真っ黒/フリーズしない）ことを確認する。異常があれば発生時刻とともに記録する
- [ ] 終業時（8 時間以上経過後）も受付画面が正常表示され、直近の受付フローが完走できることを
      確認する
- [ ] 終日を通じて、体感できるレスポンス低下（タップ〜画面遷移の遅延増大）が無いことを確認する
- [ ] 終了時刻・総経過時間・発生した異常（あれば）を記録する

**記録先**: 実施結果は本ファイル §11「UAT 記録テンプレート」の該当行、および実施の起点となる
issue #65 のコメントに、実施日・端末/iPadOS/ブラウザ・経過時間・異常有無（あれば発生時刻と
現象）を追記する。異常が見つかった場合は個別 issue を起票し、#317 の受け入れ条件（実機 soak の
手順と記録先が定義されていること）との対応関係が分かるよう本チェックリストへリンクする。

## 9. プライバシー（完了後リセット）

- [ ] 受付完了後、画面上に来訪者の氏名・会社名・要件メモが残らない
- [ ] キャンセル後も入力情報が残らない
- [ ] 受付履歴（`/admin/receptions`）に来訪者の個人情報が保存されていない（呼び出し先名・結果・所要時間のみ）

## 10. アクセシビリティ支援モード（issue #321）

常設アクセシビリティボタン（各画面の隅）から選べる 4 モード（大きな文字・ハイコントラスト・
低位置レイアウト・やさしい日本語）の実機での見え方・リーチ検証。axe/critical・serious 0 は
自動 e2e（`tests/e2e/kiosk-a11y-modes.spec.ts`）が担保済み。本節は実機でしか確認できない
「実際の見えやすさ・触れやすさ」を対象とする（実施は #65）。

### 10.1 常設ボタンの発見しやすさ・到達性

- [ ] 画面の隅（右上）のボタンが、他の操作（逃げ道バー・チャット FAB・アバター）と重ならず常に見える
- [ ] アイコン+文字が実機の反射・屋外光でも視認できる
- [ ] 待機・目的選択・情報入力・確認・呼び出し中・結果・PIN 許可待ちの各画面でボタンの位置が変わらない
- [ ] タップ領域が指（手袋着用時も想定）で確実に押せる大きさ

### 10.2 大きな文字

- [ ] 1.3倍・1.6倍で主要画面の文字が実機で明らかに大きくなったと分かる
- [ ] 1.6倍でも主要ボタン・入力欄・確認内容が画面からはみ出さず操作できる（横向き/縦向き双方）
- [ ] ソフトウェアキーボード表示時も入力欄が隠れない（拡大時は特に確認）

### 10.3 ハイコントラスト

- [ ] 直射日光下・反射の強い設置場所で、通常表示より文字・ボタンが判別しやすくなる
- [ ] ブランドロゴ・アクセント色（テナント設定時）が保持されたまま、背景と文字の区別が付く
- [ ] 背景画像（設定時）が消え、単色背景になる（意図した挙動）

### 10.4 低位置レイアウト

- [ ] 車椅子利用者・低身長者を想定した目線・リーチで、主要な選択肢/入力欄/確認ボタンに無理なく手が届く
- [ ] 画面下半分に主要操作が寄っても、上部の案内文言・ステッパーが隠れず読める
- [ ] iPad 横置き・縦置きの双方で極端なレイアウト崩れがない

### 10.5 やさしい日本語

- [ ] 主要フロー画面（待機・目的選択・検索・情報入力・確認・呼び出し中・終端）の文言が平易な表現で表示される
- [ ] bounded scope 外の画面（チェックアウト・満足度フィードバック詳細等）は既定日本語へフォールバックしても
      文脈上不自然に見えない（フォールバックが目立って破綻していない）
- [ ] ふりがな併記は本増分では未実装（follow-up）。今後の増分で対応するまでは平易な語彙・分かち書きのみで
      判読性が十分か現場の声を記録する

### 10.6 既定復帰

- [ ] 受付完了・キャンセル後の自動待機復帰で、選んだモードが全て解除され既定表示に戻る
- [ ] 無操作リセット後も同様に既定表示へ戻る（次の来訪者が前の来訪者の設定を引き継がない）
