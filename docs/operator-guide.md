# 受付運用管理者向け操作マニュアル (issue #316)

**開発者ではない受付運用管理者**が、日常運用（担当者/部署の更新・端末の追加/失効/再登録・
案内文言/アセットの変更・受付履歴の確認）と iPad 端末の設置を、**このマニュアルだけ**で
完了できることを目的とする。

障害発生時の対応（アラーム・緊急停止・手動受付への切替）は `docs/runbook.md` を見る。
本書は平時の日常運用に閉じる。

## 0. 管理画面へのログイン

- 管理画面 URL: `/admin`（例: `https://<公開ドメイン>/admin`）
- ログイン画面: `/admin/login`
- ログインには、システム管理者が発行したアカウント（管理者パスワード、または組織の SSO）
  が必要。アカウントが無い/ロックされている場合はシステム管理者（開発者）に連絡する。
- 画面左のナビゲーションから各機能へ移動する。ロールにより見える項目が異なる
  （閲覧のみの `viewer` ロールでは更新系ボタンが押せない/表示されない）。

## 1. 担当者・部署の更新（CSV 取り込み）

### 1.1 部署の一括更新

1. `/admin/departments` を開く。
2. ページ下部の **「CSV インポート」**（クリックで開閉）を開く。
3. テキストエリアに CSV を貼り付ける（1行目はヘッダ想定。既存の CSV エクスポート結果を
   雛形にすると安全）。
4. まず **「プレビュー」**ボタンを押す。「プレビュー: 新規 n 件 / 更新 n 件 / エラー n 件」
   の結果が変更されないまま表示される（この時点ではまだ反映されない）。
5. 件数・エラー内容に問題がなければ **「取り込む」**ボタンを押す。「取り込み完了: …」の
   表示に変わったら反映済み。
6. 画面上の一覧に反映結果が表示される。

API: `POST /api/admin/departments/import`（body: `{ csv, mode: "preview"|"apply" }`）。
実装: `src/app/api/admin/departments/import/route.ts`。

### 1.2 担当者の一括更新

`/admin/staff` で、上記と同じ手順（CSV インポート → プレビュー → 取り込む）を行う。

API: `POST /api/admin/staff/import`（同上の body 形式）。
実装: `src/app/api/admin/staff/import/route.ts`。

### 1.3 個別の追加・編集

CSV を使わず、画面上のフォーム/一覧からも部署・担当者を個別に追加・編集できる
（`/admin/departments` / `/admin/staff` の一覧上部・各行の編集ボタン）。少人数の変更は
個別編集、まとまった更新（人事異動シーズン等）は CSV 取り込みが効率的。

## 2. 受付端末（iPad）の追加・失効・再登録

受付端末の管理画面は 2 つある。**通常は `/admin/devices`（拠点別）を使う**
（端末の再登録=受付URL/QRの再発行までこの画面で完結するため）。

| 画面 | 用途 | 対応 API |
| --- | --- | --- |
| `/admin/devices` | 拠点（サイト）ごとの端末管理。追加・名称/設置場所/種別変更・メンテナンス表示切替・有効/無効・**受付URL(QR)の発行/再発行** | `GET/POST /api/admin/devices`, `PATCH /api/admin/devices/:id`, `POST /api/admin/devices/:id/reissue-token` |
| `/admin/kiosks` | 端末の簡易な有効/無効切替（拠点をまたいだ一覧）。`/admin/devices` の状態と自動同期する | `GET/POST /api/admin/kiosks`, `POST /api/admin/kiosks/:id/revoke`, `POST /api/admin/kiosks/:id/restore` |

### 2.1 端末の追加

1. `/admin/devices` を開く。対象の拠点（テナント/サイト）を選択する。
2. 「追加」ボタンから端末名・設置場所・種別（`kiosk` / `tablet` / `desktop`）を入力して登録する。
3. 登録直後はまだ受付URL（QR）が発行されていない状態（`token`列が「未登録」）。
   続けて 2.3「受付URL(QR)の発行」を行う。

API: `POST /api/admin/devices`（実装: `src/app/api/admin/devices/route.ts`）。

### 2.2 端末の失効（利用停止）

1. `/admin/devices` の対象端末の行で「無効化」ボタン（`data-testid="device-toggle-enabled"`。
   一覧の操作列。有効時は「無効化」、無効時は「有効化」とラベルが切り替わる）を押す。
2. 一覧の状態が「無効」に変わったことを確認する。無効化すると、その端末に対して発行済みの
   保留中の受付URLも自動的に無効になる（`PATCH /api/admin/devices/:id` に
   `{ "enabled": false }`）。
3. 再度有効化したい場合は同じボタン（トグル）で「有効化」する。

特定拠点をまたいだ一覧から素早く失効させたいだけの場合は `/admin/kiosks` の「失効」ボタン
（`POST /api/admin/kiosks/:id/revoke`）でも同等の操作ができる。

実装: `src/app/api/admin/devices/[id]/route.ts`。

### 2.3 端末の再登録（受付URL / QR の発行・再発行）

端末の初回設置時、または「iPad を機種変更した」「URL が漏えいした恐れがある」等で
**URLを作り直したい**ときに行う。

1. `/admin/devices` の対象端末の行で **「受付URL再発行」**ボタン
   （`data-testid="device-reissue"`）を押す。
2. 確認ダイアログ（`data-testid="device-reissue-dialog"`）で内容を確認し、
   **「発行する」**（`data-testid="device-reissue-confirm"`）を押す。
   — **現在有効な URL はこの操作で無効になる**（同時に複数の URL を有効にできない設計）。
3. 発行された受付URL（`https://<公開ドメイン>/kiosk/enroll?token=...`）と有効期限が
   画面に一度だけ表示される（`data-testid="device-issued-copy"` でコピー可能）。
   **この URL は再表示できないため、必ずこの場でコピー/QR化して端末側へ渡す。**
4. iPad 側で Safari を開き、この URL にアクセスする（QR コードとして印刷・表示して
   iPad のカメラで読み取らせてもよい）。有効期限内にアクセスしないと URL が失効するため、
   発行後は速やかに端末側で開く。
5. 端末側でアクセスが成功すると、その iPad がこの端末として認可され、以後は
   受付待機画面（`/kiosk`）がそのまま使える状態になる。

API: `POST /api/admin/devices/:id/reissue-token`
（実装: `src/app/api/admin/devices/[id]/reissue-token/route.ts`）。
セキュリティ: 発行された平文 URL/token はこのレスポンス以外（画面再読込・監査ログ等）には
一切残らない。**画面を閉じる前に必ず控える。**

### 2.4 一時的なメンテナンス表示

障害ではなく計画的な保守（清掃・点検等）で一時的に「メンテナンス中」表示にしたいだけの
場合は、`/admin/devices` の対象端末で「メンテナンス表示」トグル
（`data-testid="device-maintenance"`）を使う。端末自体は無効化されない
（`PATCH /api/admin/devices/:id` に `{ "maintenance": true|false }`）。

障害時の全端末停止（緊急停止モード）は `docs/runbook.md` §2 を参照。

## 3. 案内文言・アセットの変更

| 変更したいもの | 画面 | 対応 API |
| --- | --- | --- |
| ロゴ・アクセントカラー等のブランド設定 | `/admin/branding` | `GET/PUT /api/admin/branding` |
| 背景画像等のアセット（複数登録し、有効セットを切替可能） | `/admin/assets` | `GET/POST /api/admin/assets` |
| 受付フローのステップ構成・来訪者情報入力フォームの項目/ラベル | `/admin/reception-flows` | `GET/POST /api/admin/reception-flows` |
| 待機画面のサイネージ表示 | `/admin/signage` | （`/admin/signage` 画面から設定） |
| 画面遷移の演出（モーション） | `/admin/motions` | （`/admin/motions` 画面から設定） |
| 音声案内 | `/admin/voice` | （`/admin/voice` 画面から設定） |
| 表示言語設定 | `/admin/languages` | （`/admin/languages` 画面から設定） |

いずれも画面上のフォームで変更し、保存ボタンで即時反映される（受付端末側は設定を API から
取得するため、次回のポーリング/画面遷移で反映される）。ロゴ画像本体などの機微でない値は
監査ログにも記録されるが、内容（画像データそのもの）は記録しない
（`branding.updated` の監査には「ロゴ設定の有無」等のみが残る）。

## 4. 受付履歴の見方

`/admin/receptions` を開く。

- 一覧には「呼び出し結果（応答/未応答/失敗/キャンセル）・所要時間・利用端末・代替導線の
  利用有無」等の**運用ログのみ**が表示される。**来訪者の氏名・会社名などの個人情報は保存
  されていない**（システムの設計方針）。
- 画面上部の検索・フィルタで、期間・結果・端末を絞り込める。
- 「CSV エクスポート」ボタンで表示中の一覧を CSV としてダウンロードできる
  （監査ログ画面 `/admin/audit` と同じ UI 部品を使用）。

API: `GET /api/admin/receptions`（実装: `src/app/api/admin/receptions/route.ts`）。

来訪者ごとの詳細な滞在状況（入館〜退館）を見たい場合は `/admin/stay`（在館状況）、
来訪予約は `/admin/reservations` を参照する。

## 5. iPad 設置手順（運用者向け・現地作業）

開発/自動テスト観点のチェックリストは `docs/ipad-uat.md` にあるが、以下は**現地で端末を
1台設置するときの実務手順**として、必要な準備物と手順のみを抜粋・再構成したもの
（詳しい確認観点は `docs/ipad-uat.md` を参照。本書はその運用版）。

### 5.1 事前準備物

- iPad（第7世代以降を推奨）と充電ケーブル・電源アダプタ
- 現地の安定した Wi‑Fi（固定 SSID 推奨）の SSID・パスワード
- 事前に `/admin/devices` で作成しておいた端末エントリ（§2.1）と、発行直後の受付URL/QR
  （§2.3。**発行してから現地作業までの間隔が空くと有効期限切れの可能性があるため、
  現地作業の直前に発行するのが安全**）

### 5.2 設置手順

1. **電源**: 常時給電できる場所に設置する（受付完了後も画面を維持し続けるため、
   バッテリー運用は非推奨）。
2. **ネットワーク**: 設定 → Wi‑Fi で現地の SSID に接続する。
3. **エンロール（端末登録）**: Safari で §2.3 の受付URL（またはQRコード読み取り）を開く。
   成功すると、この iPad が対象端末として認可され `/kiosk`（受付待機画面）が表示される。
4. **ホーム画面への追加**: Safari の共有シートから「ホーム画面に追加」を行い、
   専用アイコンから起動できるようにする（ブラウザの chrome が表示されない standalone
   表示になる）。
5. **ガイドアクセス（キオスク固定）**: 設定 → アクセシビリティ → ガイドアクセスで、
   ホームボタン/ジェスチャによる離脱を防ぐよう固定する。画面消灯（自動ロック）は
   「なし」、または現場の運用時間に合わせて調整する。MDM を導入済みの場合は
   単一アプリモード相当の設定を確認する。
6. **アクセス制御（任意）**: 受付前に PIN 入力を必須にしたい場合は `/admin/security`
   で PIN 必須を有効化する（§0 で述べた画面。緊急停止トグルとは別の設定項目）。
   拠点のネットワークからのみ許可したい場合は IP 許可リストを設定する。
7. **動作確認**: 実際に受付フロー（目的選択 → 担当者/部署選択 → 入力 → 確認 →
   呼び出し）を最後まで一通り試す。呼び出しが担当者に届くことを確認する。
8. **管理画面からの見え方確認**: `/admin/devices` の一覧で当該端末が「有効」
   「登録済み」と表示されていること、`/admin/receptions` にテスト受付が記録されている
   ことを確認する（記録後は削除不要。テスト受付である旨をメモしておく）。

### 5.3 撤去・機種変更時

端末を撤去する、または別の iPad に置き換える場合は §2.2「端末の失効」を行う
（旧端末の URL を無効化してから、新しい端末に対して §2.3「再登録」で新しい URL を
発行する）。

## 6. 困ったときは

- 画面が反応しない・呼び出しができない等、**障害の疑いがある場合**は
  `docs/runbook.md` の手動受付切替フローに従う。
- 上記の手順で解決しない/操作方法が分からない場合は、システム管理者（開発者）に連絡する。
