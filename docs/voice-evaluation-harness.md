# 音声評価ハーネス (issue #365)

日本語音声の**精度・遅延・ターン・割り込み**をオフラインで再現可能に評価する基盤。
#369 (Transport) / #370 (STT) / #371 (TTS) / #372 (Turn) はここで定義した共通イベント形式へ
計測値を出し、変更による品質劣化を継続検知する。

本ハーネスは #319（本番受付の到達・完遂ファネル）とは責務が別である。#319 は運用 KPI、
本ハーネスは音声入力から Entity 解決までの**品質計測**を扱う。混ぜない。

## 共通イベント形式

計測は**セッション単位のイベント列**として記録する。`t` はセッション開始からの相対ミリ秒で、
**単一時計源**であること（複数クロックを混ぜると遅延が負になり、無言で 0 に丸められた指標が
緑になる）。

| イベント | 追加フィールド | 出す責務 |
| --- | --- | --- |
| `audio.onset` | — | #369 / #372 |
| `speech.end` | — | #372 |
| `stt.partial` | `text`, `stable` | #370 |
| `stt.final` | `text` | #370 |
| `turn.committed` | `text`, `trigger` | #372 |
| `entity.resolved` | `query`, `candidates[]`（score 降順） | 受付ドメイン |
| `tts.request` | `text` | #371 |
| `tts.first_byte` | — | #371 |
| `tts.playback_start` | — | #371 |
| `tts.playback_stopped` | `reason` | #371 / #372 |
| `vrm.viseme_applied` | `audioTimestampMs` | アバター側 |
| `transport.connected` | — | #369 |
| `transport.stream_open` | — | #369 |
| `transport.reconnecting` | `attempt` | #369 |
| `transport.disconnected` | `reason` | #369 |
| `transport.stats` | `droppedPackets`, `jitterMs` | #369 |
| `error` | `stage`, `code` | 全トラック |
| `session.aborted` | `stage`, `code` | 全トラック |

`error` / `session.aborted` の `code` は**列挙可能な短い識別子**に限る（64 文字上限をバリデータが
強制する）。例外メッセージをそのまま入れると PII や内部パスが混入するため。途中で失敗した実装は
必ず `session.aborted` を出すこと —— これを省いて「イベントが少ないだけ」の列を返すと、
`abortedSessionRate` に現れず失敗が緑に埋もれる。

測定起点はイベントの差分として導出する。

```text
audio.onset          → stt.partial(first)      サーバ初回 partial
audio.onset          → stt.partial(stable)     UI 確定 partial
speech.end           → turn.committed          ターン確定
turn.committed       → tts.playback_start      応答音声の開始
audio.onset(再生中)  → tts.playback_stopped    割り込み応答
tts.playback_start   → vrm.viseme_applied      音声と口形の同期誤差
```

### 設計上の約束

- **近端発話に専用イベントを作らない。** 再生区間中に落ちた `audio.onset` を近端発話として
  導出する。実装側は「割り込みらしさ」を判断する前の生の観測だけを出せばよい。
- **正解ラベルはイベントに混ぜない。** 真の割り込みか相づちかは `groundTruth` 側が持つ。
  実装は正解を知らないまま計測でき、データセットだけ差し替えられる。
- **正解は「刺激」に紐づける（観測の通番ではなく）。** 近端発話の正解は
  `nearEndStimuli: [{ id, atMs, toleranceMs, label }]` として、こちらが音を鳴らした絶対時刻で持つ。
  観測 onset はその時間窓へマッチングする。未マッチの刺激 = 検出漏れ、未マッチの観測 = 誤検出として
  **どちらも指標になる**。
  観測の通番でラベルを指すと、VAD が onset を 1 つ取りこぼしただけで以降のラベルが全てずれ、
  ターンが脱落すると注釈ごと無効化される —— つまり **provider の性能の失敗が「スキーマ違反」として
  現れて**しまい、評価ハーネスが絶対に区別すべき「実装が壊れた」と「計測が壊れた」を混同する。
- **性能の失敗をスキーマ違反にしない。** ターンの脱落・セッションの中断・onset の取りこぼしは
  すべて指標（`missedEndRate` / `abortedSessionRate` / `nearEndOnsetDetectionRate`）として現れる。
  バリデータが見るのは構造の正しさだけ。
- **計測できなかった指標は `null`。** 0 や 1 に丸めない。分母 0 を満点として緑にすると、
  計測が壊れた回帰を検知できなくなる。SLO 判定は `null` を `skipped` として扱い、
  `strict: true` のプロファイル（`uat`）では計測不能自体を違反にする。
- **生音声・音声 URI をイベントに載せない。** `audioUri` 等のフィールドはバリデータが弾く。

### 停止の帰属

「どの近端発話が再生を止めたか」は `attributeBargeInStops` の**1 箇所だけ**で決め、遅延指標と
誤停止率の両方がそれを使う。別々の規則で帰属すると指標同士が矛盾する。

規則は「停止より `MIN_BARGE_IN_REACTION_MS`(30ms) 以上前にある onset のうち最も遅いもの」。
単純な「区間の最初」だと、相づちの直後に本当の割り込みが来た系列で相づちが原因にされ、
「停止直前」だと、割り込みで停止判断が飛んでいる最中に入った相づちが原因にされる。

### #369〜#372 の適合の示し方

各 issue の実装は、自分のテストで次を通すこと。これが適合ゲートになる。

```ts
const validation = validateVoiceEvalSession(session);
expect(validation.errors).toEqual([]);
```

`runVoiceEvalSuite` は provider ごとに `passed`（SLO + スキーマ違反 + 実行エラー）を返す。
`slo.passed` だけを見ないこと —— セッションが検証で落ちて 1 件も計測されなかった provider は
「違反 0 件」になりうる。

## 指標

| 分類 | 指標 |
| --- | --- |
| STT | 発話ごと CER、**コーパス CER**（総編集距離/総文字数）、人名/部門名の一致率 |
| Entity | Top1 / Top3 包含率、Recall、Precision |
| ターン | 誤終了率、終了見逃し率、フィラー起因の誤応答率 |
| 割り込み | 真の割り込み検出率、相づち/エコー誤停止率、誤停止率全体、onset 検出率、誤検出件数 |
| 信頼性 | 中断セッション率、エラー件数/セッション、再接続数/セッション、jitter |
| 遅延 | 上表の各起点の P50 / P95 / max |

CER が小さくても人名だけ落ちる回帰（同音異字の取り違え）は受付として致命的なので、
固有名詞の一致率を独立指標として持つ。実測でも、同音異字を注入した provider は
コーパス CER 2.4% のまま人名一致率が 33% まで落ちる。

CER は**コーパス CER を主指標**にする。発話ごと CER の中央値は、固有名詞を含まない短い発話が
多数を占めると 0 に張り付き、回帰を隠してしまう。

集計は**割合の平均ではなく分子分母の合算**で行う。1 ターンのセッションと 9 ターンのセッションが
混ざっても重み付けが狂わない（率の平均だと 0.5、正しい pooling だと 0.1 になる場面がある）。

## プロファイルと SLO

| | `ci` | `uat` |
| --- | --- | --- |
| 用途 | 合成 fixture の軽量セット（毎回のローカル品質ゲート） | 実機・実 provider の完全セット (#65) |
| 計測不能の扱い (`strict`) | skipped | **違反** |
| UI 確定 partial P50 | 600ms | **300ms** |
| 割り込み → 停止 P50 / P95 | 300 / 600ms | **150 / 300ms** |
| 短答 speech end → first audio P50 | 1000ms | **500ms** |
| 自由発話 speech end → first audio P50 | 1800ms | **900ms** |
| viseme 同期誤差 P50 | 100ms | **50ms** |
| 誤割り込み率（上限） | 5% | **2%** |
| 真の割り込み検出率（下限） | 90% | **98%** |
| 近端 onset 検出率（下限） | 85% | **95%** |
| 誤ターン終了率（上限） | 6% | **3%** |
| ターン終了見逃し率（上限） | 6% | **3%** |
| コーパス CER（上限） | 10% | **5%** |
| 人名一致率（下限） | 90% | **98%** |
| 部門名一致率（下限） | 90% | **95%** |
| 担当者 Top3 包含率（下限） | 95% | **99%** |
| 中断セッション率（上限） | 0 | **0** |

**SLO セットは対称に保つこと。** 誤停止・誤終了（false positive）だけに上限を置いて、検出漏れ・
終了見逃し（false negative）に下限を置かないと、「何もしない provider」が最も安く緑になる。
実際、割り込みを一切止めない provider は誤停止率 0 で上限だけの SLO を通過する。
上限を足すときは対になる下限も足す。

`uat` 列が issue #365 の初期 SLO 案（検出率・見逃し率など対を成す項目は本 PR で補った）。
**実機検証で更新する前提の暫定値**であり、更新時は issue #365 と本ドキュメントと
`evaluation-thresholds.ts` を併せて直す。

`ci` を緩めてあるのは、合成データのタイミングノイズで恒常的に赤くしないため。`ci` は
絶対値の保証ではなく**回帰の方向**を見るためのもの。

「発信前確認後の誤担当者発信 0 件」はオフラインのイベント列だけでは判定できない
（実際の発信操作を伴うため）。#65 の実機 UAT で確認する項目として残す。

## データの取り扱い

- **実音声はリポジトリに入れない。** 既定のデータセットは合成発話テキストと正解ラベルのみ。
- 実音声・協力者音声を使う場合は **明示的同意・匿名化・ライセンス**を必須とする
  （`docs/license-privacy-guide.md` / #105 に従う）。
- 実機セッションで得た音声・書き起こしはリポジトリ外に置き、**評価完了後 30 日以内に削除**する。
  リポジトリへ持ち込んでよいのは計測イベントと集計値だけで、`stt.final` の生テキストを
  含むセッションは持ち込まない（レポート出力にも生の書き起こしは含めない）。
- レポート（JSON / CSV / Markdown）は集計値のみを含む。共有前に人名が混じっていないことを
  確認する。

## MOS 手動評価プロトコル

自然性・聞き取りやすさは自動計測できないため、手動評価として分ける。

1. 評価者は 5 名以上、受付業務の関係者に限定しない（耳の慣れを避ける）。
2. 各評価者に同一の 10 サンプル（担当者名 4 / 部門名 3 / 用件 3）を**順序をランダム化**して提示。
3. 5 段階で 2 軸を採点する。
   - 自然性: 5 = 人の発話と区別しにくい / 1 = 明らかに機械的で不快
   - 聞き取りやすさ: 5 = 一度で聞き取れる / 1 = 聞き返しが必要
4. provider 名・設定は評価者に伏せる（ブラインド）。
5. 平均と標準偏差を記録し、**人名サンプルは別集計**にする（人名の発音誤りは平均に埋もれる）。
6. 目標: 自然性 3.5 以上、聞き取りやすさ 4.0 以上。人名の発音正解率 95% 以上。

## #65（iPad 実機 UAT）への手順

1. 対象ビルドを dev へデプロイし、kiosk を実機で開く。
2. `VOICE_EVAL_PROFILE=uat` で実 provider を差した `VoiceEvalProvider` を登録する
   （実装は #370 / #371 / #372 側）。
3. `tests/voice-evaluation/dataset.ts` と同じ切り口の発話を、**協力者が実機に向けて**発話する。
   同意取得を先に行い、録音するかどうかを明示する。
4. セッションのイベント列を書き出し、`runVoiceEvalSuite` に流して Markdown レポートを得る。
5. レポートの SLO 違反と `skipped`（計測不能）を確認する。`skipped` が残る場合は計測配線の
   欠落であり、SLO 充足とは扱わない。
6. 実測に基づき `uat` プロファイルの閾値を更新し、issue #365 の初期 SLO 案を改訂する。
7. 音声・書き起こしは上記の保存/削除ポリシーに従って破棄する。
